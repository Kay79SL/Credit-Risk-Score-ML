◆ Variable transformations — what, why, and where
Feature engineering cell

EDUCATION & MARRIAGE — code recoding

EDUCATION codes 0, 5, 6 collapsed to 4 ("Other"). MARRIAGE code 0 collapsed to 3 ("Other"). These values are not defined in the original dataset description. Done before encoding to prevent spurious dummy variables being created from junk codes.

Feature engineering cell

PAY_0 → PAY_0_BIN (binning)

PAY_0 clipped from range (−2 to 8+) to (−2 to 4) using .clip(lower=−2, upper=4). Values ≥ 4 collapsed into a single "4+" category. Used for EDA visualisation only — original PAY_0 retained for modelling to avoid information loss.

Feature engineering cell

PAY_RATIO1–6 (computed ratio)

PAY_AMTi / (|BILL_AMTi| + 1). Absolute value prevents division issues with negative bill amounts (credits). +1 avoids division by zero when BILL_AMT is zero. Produces a continuous 0–∞ feature representing monthly repayment fraction.

Cell 62 — post feature engineering

PAY_RATIO1–6 & PAY_RATIO_6M_MEAN — clipping at 5

Applied .clip(upper=5) to all 7 payment ratio columns. When BILL_AMT is near-zero, even a small payment produces extremely large ratios (e.g. 10,000+) which distort model training. Clipping at 5 caps outliers while preserving the full meaningful range (0 = no payment, 1 = full payment, 5+ = overpayment).

Feature engineering cell

BILL_AMT — clipped at 0 for utilization

When computing UTILIZATION6M_MEAN, billing amounts are clipped at 0 using .clip(lower=0) before averaging. Negative BILL_AMT values represent credit balances (money owed back to the customer) and should not reduce the utilization estimate.

Preprocessing pipeline cell

Numeric features — StandardScaler

All 30 numeric columns scaled to zero mean and unit variance. Applied inside the sklearn Pipeline — fitted on training data only and applied to test set via .transform(). Prevents data leakage. Essential for Logistic Regression (distance-sensitive) and PCA. Tree-based models are scale-invariant but included for pipeline consistency.

Preprocessing pipeline cell

Categorical features — OneHotEncoder

SEX, EDUCATION, MARRIAGE converted to binary dummy columns via OneHotEncoder(handle_unknown="ignore"). Applied inside the same ColumnTransformer pipeline. handle_unknown="ignore" prevents errors if unseen categories appear at inference time. After OHE, the pipeline output expands from 33 input features to a larger numeric matrix.

Preprocessing pipeline cell

Missing value imputation — SimpleImputer

Numeric features: SimpleImputer(strategy="median"). Categorical features: SimpleImputer(strategy="most_frequent"). No missing values were found in this dataset, but the imputer is included as a safeguard for inference-time data that may have gaps. Fitted on training data only.

PCA cell (numeric features only)

Numeric features — PCA dimensionality reduction

StandardScaler applied separately to the 30 numeric columns. PCA fitted with n_components=30, then truncated to retain 95% cumulative explained variance (24 components, PC1=26.7%, PC2=15.4%). Applied to LR_PCA model only. Result: AUC dropped to 0.7052 — PCA compressed the dominant PAY_0 signal, confirming dimensionality reduction is not beneficial here.

Skewness — what was done, where, and how it affects the models No log transformation or Box-Cox applied to BILL_AMT or PAY_AMT directly. The skew is addressed through three mechanisms applied at different stages:

(1) StandardScaler — applied in Step 10 (Preprocessing Pipeline), fitted on X_train only.
StandardScaler subtracts the column mean and divides by the standard deviation, producing features with mean=0 and std=1. It does not remove skew — a right-skewed distribution stays right-skewed, just centred and rescaled. What it does do is equalise the magnitude of all numeric features so that no single column dominates the LR loss function simply because its values are large. Without scaling, BILL_AMT (range 0–983,000) and AGE (range 21–79) would contribute vastly different raw numbers to the log-odds sum — LR's gradient descent would chase the large-magnitude feature and produce tiny, unstable coefficients for the rest. After StandardScaler, all features are on the same ≈ −3 to +3 scale, coefficients are numerically stable, and the optimiser converges faster and more reliably.

(2) PAY_RATIO features — applied in Step 8 (Feature Engineering), before the split.
This is more than just a scaling fix — it is a fundamentally different representation of the data. Raw BILL_AMT tells LR the size of the debt; raw PAY_AMT tells LR the size of the payment. Neither alone tells LR the relationship between the two — which is the actual credit risk signal. PAY_RATIO = PAY_AMT / (|BILL_AMT| + 1) compresses both into one bounded feature that directly encodes the behavioural question: what fraction of what was owed got paid back?
Consider two customers both with BILL_AMT = 50,000: Customer A paid 49,500 (PAY_RATIO ≈ 0.99); Customer B paid 500 (PAY_RATIO ≈ 0.01). After StandardScaler, both have identical scaled BILL_AMT. Only PAY_RATIO distinguishes them. LR can assign a single clear coefficient to PAY_RATIO: high ratio → low default risk, low ratio → high default risk. The relationship is monotonic and approximately linear in log-odds space, which is exactly what LR is designed to exploit. After clipping at 5, PAY_RATIO ranges 0–5 — already close to the ≈ −3 to +3 range StandardScaler will produce, so it arrives at the pipeline in a stable, well-conditioned state. The raw BILL_AMT and PAY_AMT columns are retained alongside PAY_RATIO because they carry complementary information (total exposure level, absolute repayment capacity) that the ratio alone does not convey.

(3) UTILIZATION6M_MEAN — applied in Step 8, before the split.
Divides the 6-month average bill by credit limit, producing a 0–1+ bounded ratio that is far less skewed than raw BILL_AMT and directly interpretable as credit usage pressure. Same logic as PAY_RATIO — the ratio captures the meaningful relationship that the raw values cannot.

Tree-based models (RF, XGBoost): unaffected by any of the above. They make splits on ordinal thresholds — whether BILL_AMT is 50,000 or 50 is irrelevant as long as the rank order is preserved. Skew, scale, and magnitude do not affect split quality.
7. EDA — correlation analysis (Pearson & Spearman) Done
Both Pearson and Spearman computed. PAY_0–PAY_6 dominate both. PAY_AMT1–6 show weak Pearson (~0.06) but moderate Spearman (~0.13–0.16) — indicates non-linear monotonic relationship that Pearson misses. LIMIT_BAL Pearson=0.154, Spearman=0.170. BILL_AMT columns weak and negative (high bills alone do not predict default; payment behaviour matters more).
Multicollinearity — variables with high inter-correlation Several variable groups are strongly correlated with each other, not just with the target. Key multicollinearity concerns identified:
  • BILL_AMT1–6: Highly intercorrelated (r ≈ 0.85–0.95 between adjacent months). A customer's outstanding balance changes slowly, so consecutive months are nearly identical. This is the strongest collinearity in the dataset. Addressed by creating BILL_TOTAL_6M (aggregate) and UTILIZATION6M_MEAN (ratio) as summary features, reducing the need to include all 6 raw columns individually.
  • PAY_AMT1–6: Moderately intercorrelated (r ≈ 0.50–0.70). Customers who pay well in one month tend to pay well in others. Addressed by creating PAY_TOTAL_6M and PAY_RATIO_6M_MEAN as summaries.
  • PAY_0 and PAY_0_BIN: Near-perfect correlation by construction (PAY_0_BIN is a clipped version of PAY_0). Decision: PAY_0_BIN dropped from the modelling dataset — PAY_0 retained.
  • PAY_RATIO_6M_MEAN and PAY_RATIOi: PAY_RATIO_6M_MEAN is the arithmetic mean of PAY_RATIO1–6, so it is collinear with each individual ratio. Both included deliberately — the individual ratios carry month-specific timing information; the mean carries the 6-month summary signal. For LR this creates mild redundancy; tree-based models handle it via feature selection at each split.
  • BILL_TOTAL_6M and BILL_AMTi: BILL_TOTAL_6M is the sum of BILL_AMT1–6, so it is linearly dependent on them. Acceptable because BILL_TOTAL_6M replaces rather than supplements the individual columns in the intended modelling signal.
  • AVG_DELAY_6M and PAY_0–PAY_6: AVG_DELAY_6M is the mean of the six PAY_X columns, so it is collinear with each. Again, the aggregate captures overall delinquency level while individual columns capture timing — both retained intentionally.
Impact on models: Logistic Regression is most sensitive to multicollinearity (inflated standard errors, unstable coefficients). However since prediction rather than coefficient interpretation is the primary goal, and since class_weight="balanced" and regularisation (C parameter) are applied, multicollinearity does not invalidate the LR model — it just makes individual coefficient values less interpretable. Random Forest and XGBoost are largely unaffected by multicollinearity as they use greedy feature selection at each split.
8. Feature engineering — 10 new features Done
  • PAY_RATIO1–6: PAY_AMTi / (|BILL_AMTi| + 1) — monthly repayment fraction
  • UTILIZATION6M_MEAN: mean(BILL_AMT1–6 clipped ≥ 0) / (LIMIT_BAL + 1) — credit usage
  • BILL_TOTAL_6M: sum of BILL_AMT1–6 — total 6-month exposure
  • PAY_TOTAL_6M: sum of PAY_AMT1–6 — total 6-month repayments
  • PAY_RATIO_6M_MEAN: mean of PAY_RATIO1–6 — summary repayment signal
  • LATE_COUNT_6M: count of months where PAY_X > 0
  • MAX_DELAY_6M: max delay month across PAY_0–PAY_6
  • AVG_DELAY_6M: mean delay across PAY_0–PAY_6
  • PAY_GAP_6M_TOTAL: BILL_TOTAL_6M − PAY_TOTAL_6M (net debt accumulation)
Transformations applied here — yes, these are the same transformations listed in the "Variable Transformations" section aboveThe dedicated transformations section at the top of this document lists every transform across the entire pipeline for easy reference. This step is where the following were first applied to the data: EDUCATION/MARRIAGE recoding (0,5,6→4 and 0→3); PAY_0_BIN creation via clip(lower=−2, upper=4); BILL_AMT clipping at 0 for the UTILIZATION feature; the +1 denominator adjustment in PAY_RATIO to prevent division by zero. All are within-row arithmetic only — no cross-row aggregation — which is why applying these before the train/test split is valid and does not cause data leakage. The StandardScaler, OneHotEncoder, and imputation are NOT done here — those happen in Step 10 (preprocessing pipeline), fitted on training data only.
9. PAY_RATIO clipping at 5 Done
A separate post-engineering step applied to all 7 payment ratio columns: PAY_RATIO1 through PAY_RATIO6 and PAY_RATIO_6M_MEAN. The .clip(upper=5) operation was applied to the full dataset (df_fe) before the train/test split.
Outlier — extreme PAY_RATIO values and why they arise PAY_RATIO is computed as PAY_AMT / (|BILL_AMT| + 1). The denominator (+1) prevents division by zero when BILL_AMT = 0, but it creates a different problem: when BILL_AMT is legitimately very small (e.g. 1 or 2 NTD), even a modest payment produces a very large ratio. Examples from the data: BILL_AMT=1, PAY_AMT=5,000 → ratio = 2,500. BILL_AMT=0, PAY_AMT=3,000 → ratio = 3,000. This happens in months where a customer has paid off their entire balance and the new statement has not yet arrived — a completely normal situation.

These values are not errors, but they behave like outliers in the model: they dominate the feature range, compress all other values toward zero in relative terms, and distort gradient calculations in Logistic Regression and XGBoost, and threshold selection in tree-based models.

Why clip at 5 specifically: The value 5 was chosen to preserve the full interpretable range — a ratio of 1.0 means the customer paid exactly their bill; 2.0 means they paid double; 5.0 means they paid five times the outstanding amount. Beyond 5, the risk signal is identical (the customer is not in financial distress regardless of the exact multiple). The threshold of 5 was determined by domain reasoning rather than a formal percentile cutoff, but it is consistent with credit risk industry practice of capping repayment ratios at 3–5× for behavioural scoring.

Columns affected: PAY_RATIO1, PAY_RATIO2, PAY_RATIO3, PAY_RATIO4, PAY_RATIO5, PAY_RATIO6, PAY_RATIO_6M_MEAN — all 7 ratio columns.
Decision: Clipped at 5. Raw values above 5 replaced with 5. No rows removed.

17. Final model selection — based on DeLong results To do
Once DeLong pairwise results are available, use the p-values to make a statistically justified final model choice using the following decision logic:
  • If XGB_Tuned vs RF_Tuned p < 0.05: The AUC difference is statistically significant — select XGB_Tuned as the single best model based on higher AUC (0.7811) and better recall on defaulters (0.6172). Report DeLong z-stat and p-value in thesis.
  • If XGB_Tuned vs RF_Tuned p ≥ 0.05: AUC difference is not significant (consistent with Nemenyi p=1.000). Both models are statistically equivalent. Final selection then defaults to business criteria: (a) if recall on defaulters is the priority → XGB_Tuned (0.6172 vs 0.5855); (b) if precision and F1 are the priority → RF_Tuned (F1=0.5409, Precision=0.5026). Document the selection rationale explicitly in the thesis.
  • If any other model pair shows p < 0.05: Update the model comparison narrative to reflect which AUC gaps are real vs noise. This may affect which tier groupings are reported.
Why this step mattersWithout DeLong, choosing XGB_Tuned over RF_Tuned rests only on a 0.0046 AUC difference which may be within sampling noise from the single 6,000-row test set. DeLong provides the statistical backing to either confirm or deny that difference. This is standard practice in clinical and financial ML model selection — the selected model must be defensible not just by the number but by a p-value. Reference in thesis: DeLong et al. (1988), Dietterich (1998).

◆ Key decisions made

📊 Primary metric: ROC-AUC not accuracy

22.12% default rate makes accuracy misleading — always predicting no-default scores 77.88%. ROC-AUC measures ranking ability across all thresholds, directly relevant to credit scoring.

⚖️ Class imbalance corrections

class_weight="balanced" for LR, DT, RF. scale_pos_weight=3.52 (23,364/6,636) for XGBoost. Stratified 80/20 split preserves 78/22 class ratio in both sets.

🔢 Categorical encoding: OneHotEncoder

SEX, EDUCATION, MARRIAGE converted to binary dummies inside the sklearn Pipeline. Undocumented codes recoded first. handle_unknown="ignore" protects inference-time robustness.

📏 Numeric scaling: StandardScaler

Fitted on training data only, applied to test via Pipeline. Essential for LR and PCA. Tree-based models scale-invariant but included for pipeline consistency and fair comparison.

🗑️ PAY_0_BIN dropped from modelling

Created for EDA visualisation only (clipped −2 to 4). Original PAY_0 retained for modelling to avoid information loss and multicollinearity with the continuous PAY_0 feature.

✂️ PAY_RATIO clipped at 5

Extreme ratios (up to thousands) when BILL_AMT ≈ 0 distort model training. Capping at 5 removes outlier distortion while preserving the full meaningful range of the feature.

🔬 Friedman + Nemenyi, not just a ranking table

Descriptive AUC ranking is insufficient academically. Friedman (χ²=32.44, p<0.001) + Nemenyi post-hoc (Demšar, 2006) provide inferential statistical evidence. XGB_Tuned and RF_Tuned statistically equivalent (p=1.000).

🏆 XGB_Tuned selected for PD scoring

Highest ROC-AUC (0.7811), best recall on defaulters (0.6172). Decision bands: Approve (PD<0.20), Review (0.20–0.50), Reject (>0.50). Statistically indistinguishable from RF_Tuned.

◆ Executive summary

The PD modelling pipeline is substantially complete. The dataset is clean (no missing values, no duplicates) but exhibits a 22.12% default rate requiring explicit imbalance correction across all models. Outlier investigation identified one extreme LIMIT_BAL customer (1,000,000) and near-complete separation in rare PAY_0 categories — both retained in the dataset after review. Key transformations applied include: categorical recoding of undocumented EDUCATION and MARRIAGE codes, PAY_RATIO clipping at 5 to control extreme outliers, BILL_AMT clipping at 0 for utilization, StandardScaler on all numeric features (train-fitted only), and OneHotEncoder on categorical features inside a leakage-free Pipeline.

The strongest predictor is PAY_0 (r=0.325) with a clear risk gradient (12.8% on-time → 75.8% at 3 months late). Ten engineered features were added. Nine models were trained. XGB_Tuned (AUC=0.7811) and RF_Tuned (AUC=0.7765) are joint best — statistically indistinguishable (Nemenyi p=1.000), both significantly outperforming LR (p=0.003, Friedman χ²=32.44). GridSearchCV produced zero improvement for LR or DT. PCA+LR performed worst (AUC=0.7052), confirming the feature space is well-structured and dimensionality reduction is not beneficial here. Remaining: DeLong testing, SHAP explainability, and governance documentation.
